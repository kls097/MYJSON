## 🐛 Bug 分类汇总

### Bug #1: 末尾多余逗号 (Trailing Comma) — **影响 16/24 个测试**

**严重程度: 🔴 严重**

**现象:** 当一侧有属性被删除/新增时，对齐算法在生成另一侧的内容时，最后一个有效属性/元素末尾会多出一个逗号（trailing comma），导致提取出的 JSON 不合法。

**举例:**

左侧原始 JSON:
```json
{
  "name": "test",
  "value": 123
}
```

右侧增加了属性后，左侧对齐内容变为:
```json
{
  "name": "test",
  "value": 123,     ← 多余的逗号！原本是最后一个属性，不该有逗号
                     ← 占位空行（对应右侧新增的属性）
}
```

这个逗号是因为对齐算法在 `alignObjectProps` 函数中计算 `isLast` 时，是基于 `allKeys`（左右合并后的所有键）来判断的，而不是基于当前侧实际存在的键。当后面的键只存在于另一侧时，当前侧最后一个真实属性被错误地添加了逗号。

**根因定位:**

`jsonComparer.js` → `alignObjectProps()` 函数，约第 508-570 行：

```javascript
const isLast = i === allKeys.length - 1  // ❌ 基于合并后的所有键判断
const comma = isLast ? '' : ','           // 导致非末尾键都加逗号，包括占位行前的最后一个真实属性
```

同样的问题存在于 `alignArrays()` 函数中数组元素的逗号处理。

**影响的测试用例:** #1, #2, #3, #6, #8, #9, #10, #12, #13, #14, #15, #18, #19, #20, #22, #24

### Bug #2: 键顺序改变导致语义不一致 — **影响 3/24 个测试**

**严重程度: 🟡 中等**

**现象:** 当左右两侧 JSON 的键存在顺序差异时，对齐算法会按左侧顺序重新排列，导致提取后的右侧 JSON 键顺序与原始输入不一致（语义等价但顺序变了）。

**影响的测试用例:** #10, #19, #23

**说明:** 这个不影响 JSON 解析的合法性，但如果用户依赖键顺序（比如做二次比较），会产生困惑。

---

### Bug #3: 数组对齐时占位行逗号传递错误 — **影响 4/24 个测试**

**严重程度: 🔴 严重**

**现象:** 在数组场景中，当数组元素被删除时，前一个元素的闭合括号 `}` 会带上逗号 `,`，但后面全是占位空行直到 `]`，提取后变成:

```json
{
  "items": [
    { ... },    ← 逗号正确
    { ... },    ← 多余逗号！后面的元素在本侧不存在
  ]
}
```

**影响的测试用例:** #3, #6, #14, #22

## 🔧 修复建议

### 核心修复: `alignObjectProps()` 的逗号逻辑

当前的 `isLast` 判断基于合并后的 `allKeys` 索引，应改为：**对每一侧分别计算该侧最后一个有效属性的位置**。

```javascript
// ❌ 当前逻辑
const isLast = i === allKeys.length - 1
const comma = isLast ? '' : ','

// ✅ 修复方案: 分别计算左右侧的 isLast
// 找到左侧最后一个存在的键的索引
const lastLeftKeyIndex = findLastIndex(allKeys, k => k in leftObj)
const lastRightKeyIndex = findLastIndex(allKeys, k => k in rightObj)

// 对于左侧的行: 逗号由左侧自己的最后一个键决定
const leftComma = (hasLeft && i === lastLeftKeyIndex) ? '' : ','
// 对于右侧的行: 逗号由右侧自己的最后一个键决定  
const rightComma = (hasRight && i === lastRightKeyIndex) ? '' : ','
```

### 同样修复 `alignArrays()` 的逗号逻辑

数组中也需要分别为左右两侧计算是否为最后一个有效元素。

### 键顺序问题（可选修复）

`alignObjectProps` 中的 `allKeys.sort()` 优先按左侧顺序排列，新增键追加到末尾。如果需要保持右侧原始顺序，可以改为：只排序新增键，保持两侧各自原始顺序。

## 📝 结论

JSON 比较功能的**核心 bug 是逗号处理逻辑错误**，影响了 75% 的测试场景。根因是对齐算法在生成对齐内容时，逗号的添加基于"合并后的所有键/元素"的位置判断，而没有考虑到每一侧实际存在的键/元素是不同的。这导致提取出的非占位行拼接后不是合法的 JSON，用户在编辑器中看到的内容也会出现格式错误提示。

修复后建议重跑本测试脚本验证，目标是 24/24 全部通过。
